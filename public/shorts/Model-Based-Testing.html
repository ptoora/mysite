Model Based Testing

Your state model represents a set of transitions taking the domain from one state to another. Tradidional approaches for drawing out test cases center on happy and sad path behaviour.
We can use model based testing to identify additional test cases that are more likely to manifest bugs because it's a more complex aspect of the application.
Model based testing validates that our application honours the state model. In order to do this we need to carefully understand the various states, which may not be explicitly described anywhere.

My approach is to draw out these test cases by:

  • Identify the states
  • Identify the input actions
  • Combine the two to form tests

Here's a simple example of a login model. There are four states:

  • Logged in
  • Logged in
  • First time logged in
  • Expired login

There are five input actions:

  • Login
  • Go to a public page
  • Go to a restricted page
  • Logout
  • Invalid login entry

Combine the two to form twenty scenarios. Test cases you are unlikely to have thought about will manifest, as if by magic, and these have a good chance of exposing bugs. Eg. Go to a restricted page when the user is logged out.

Hopefully a good step to becoming a 'model tester'.
