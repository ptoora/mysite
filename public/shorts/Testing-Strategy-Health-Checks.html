Testing the Test Strategy

Test Strategies are traditionally expressed in a generalised form. These die an early death, unread, neglected, lost in an ocean of project documentation.
Why? because they have no relevance, are abstract, have no commitment associated. The process of defining them is vague; who writes it, what does it contain, do we even need it.

Testing implementations vary by numerous factors :

  • Front end apps
  • Back end APIs
  • Native mobile app
  • Web monolith
  • Third party supplier
  • Team resources
  • Disposition to risk
  • Architecture
  • New or legacy
  • Ease of fallback
Eg. For some, time to market is king, so streamlined testing with easy fallback and testing in production is an approach. For a financial or health application, testing will be significantly risk averse and so will favour much more oversight.

Drawing from a generic best practice template, we should build individual project quality health checks that define the particular approach measured against best practice. Where a gap exists, remedial actions should be defined. The health check should inclue;
  • sit in same code repository,
  • be agreed by the team,
  • be maintained by the QA,
  • and adhoc audits should be run to check its veracity, rendering it a testable artifact.
