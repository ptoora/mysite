Parsh's Testing Wiki - Definition Of Ready

The 'Definition Of Ready (DOR)' is more important than the 'Definition Of Done' because it sets the foundation of the SDLC for the Story. It is the product of an Agile Refinement and involves; the Business, the Developer and the QA. Before a Story is ready for having code developed, it should pass the five DOR tests:

  1. Is there business value?
    Does it meet a priority business need and is it a step in your MVP.
  2. Is it comprehensive?
    Has it considered most new use-cases and addressed the impact on existing behaviour. Have we specificed the Acceptance Criteria for the APIs in a separate but related Story, and do they expose contracts. Have we given consideration to; performance, security, seo, devops (logging), data analytics, and accessibility.
  3. Is it accurate?
    The Acceptance Criteria should be unambiguous, structured (following Gherkin format), with well defined scope
  4. Is is testable via automation?
    Can we automate easily, do we have good quality test data, and do we have available testing environments
  5. Is it achievable
    Has it been sized and is comfortably deliverable (built and tested) within the sprint/kanban timescales.

Eg Let's take an example of a Story which requires that only authorised people can access an account.

  1. Is there business value?
    Our most important use case might be users on our android app, so we might just focus on the Login Story for that, leaving other requirements (eg. Web, online password reset) for later.
  2. Is it comprehensive?
    Perhaps we need to include session expiry in the Story as an important security measure. Perhaps we need to clarify what part of the account the user can see, and describe the error the user gets with an invalid login. Additonal ACs might be needed on the APIs to catch hackers bypassing the UI validation. Also we may want to add performance testing to ensure a good user experience.
  3. Is it accurate?
    We should describe; the context, the action, the behaviour, and the result of the action in each Acceptance Criteria, using a Gherkin format of 'Given, When, Then'.
  4. Is is testable via automation?
    Are there reliable html attributes, such that UI automation can proceed. Do we have stubs for APIs so we can progress and test independently.
  5. Is it Achievable
    The Story should be no more than five Story points, otherwise consider breaking it down into smaller chunks.
